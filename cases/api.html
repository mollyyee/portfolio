Designing to drive company-wide momentum to transform API quality

PROJECT CONTEXT

With no API standard, building delightful cross-product experiences at scale are near impossible.

As VMware transitions to SaaS, user experience is becoming critical to our business. In order to build cross-product experiences, internal teams need to be able to call each other’s APIs. However, a lack of API standards, consistent documentation and analytics for our wide set of APIs that made this extremely difficult.

THE SOLUTION

First, we needed to create visibility around the state of our APIs.

We built analytics and scoring for all the APIs. Without proper analytics, it was impossible for us to understand how teams were doing, where we needed improvements, and which APIs were most important to customers.

I designed a leaderboard to display scoring for API quality across our 100 products. We didn’t have any official “authority” to improve API quality. A leaderboard was a soft influence approach to quickly provide value and visibility to executives who did have that authority.

We gave leadership a way to definitively evaluate the state of APIs at VMware.
As a small team of seven in a company of 30,000, we relied on soft influence to improve API quality. A leaderboard was a fun way to reward well-crafted APIs. Top quality teams received medals for holding the best scores!

Then we needed to prevent problematic APIs from going out.

On top of fixing existing APIs to improve quality, engineer architects conducted reviews to prevent low quality APIs from being shipped. Some of these architects were responsible for dozens of microservices. It was difficult to manage quality. I designed the change logs that helped architects quickly find “problem APIs.”

Project Impact

When teams saw the analytics, they created Jira tickets with our data to begin logging issues to improve their API quality. Our strategy was working! Teams were getting the visibility they needed into their API health and taking action to fix issues.

Next Steps

We want to continue adding more useful data to our analytics platform. We want to include Google Analytics information and telemetry data to give product teams a sense of usage. This way we can add a layer of criticality on top of our data and begin prioritizing APIs by how frequently customers use them.
